# Mini Social Network

A production-shaped backend for a small social network: registration with email
verification, JWT auth, posts, likes, follows, a personalized feed, search/pagination,
and a Celery-scheduled cleanup job вҖ” built with FastAPI and async SQLAlchemy over
PostgreSQL.

The full architectural rationale (why each layer/technology/pattern was chosen, and
the trade-offs considered) lives in the accompanying design dossier; this README is
the practical "get it running" reference.

## Tech stack

| Layer | Choice | Why |
|---|---|---|
| API | FastAPI (async) | Native async, Pydantic-based validation, DI maps onto layered architecture |
| ORM | SQLAlchemy 2.0 (async) | Typed models, explicit eager-loading, no event-loop blocking |
| Database | PostgreSQL 16 | Real constraints/indexes, `asyncpg` driver |
| Migrations | Alembic | Versioned, autogenerated, revertible schema changes |
| Cache/broker | Redis | Celery broker + result backend |
| Background jobs | Celery + Celery Beat | Async email send, scheduled token cleanup |
| Auth | JWT (`pyjwt`) + `passlib`/`bcrypt` | Stateless auth, slow password hashing |
| Validation | Pydantic v2 | Wire-boundary validation, `EmailStr`, `Annotated` constraints |
| Testing | pytest + pytest-asyncio + httpx | Async integration tests against a real Postgres test DB |
| Lint/format | Ruff + Black | Fast linting, zero-config formatting |
| Containers | Docker + Docker Compose | `app`, `db`, `redis`, `celery-worker`, `celery-beat` |

## Architecture

```
API (routers) -> Service (business rules, permissions, transactions) -> Repository (SQL) -> Database
```

Routers know HTTP, not SQL. Services know business rules, not SQL or HTTP. Repositories
know SQL for one aggregate, not business rules. Dependencies always point inward вҖ”
`app/api` is the only layer allowed to import FastAPI.

```
  в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ1в”Җв”Җв”ҖNв”Җв”Җ verification_tokens     в”Ӯ

  в””в”Җв”ҖNв”Җв”Җв”ҖNв”Җв”Җ follows (follower_id, followee_id) в”Җв”Җв”ҳ
```

- `users`: `UNIQUE(username)`, `UNIQUE(email)`
- `posts`: `FK author_id -> users.id ON DELETE CASCADE`, `CHECK(1 <= len(content) <= 500)`, index on `(author_id, created_at)`
- `likes`: `UNIQUE(user_id, post_id)` вҖ” the actual race-condition defense for double-likes
- `follows`: composite `PK(follower_id, followee_id)`, `CHECK(follower_id <> followee_id)`
- `verification_tokens`: `UNIQUE(token_hash)`, index on `expires_at` (used by the cleanup job)

## Installation

```bash
git clone <this-repo>
cd socialmedia
cp .env.example .env
docker compose up --build
```

That's it вҖ” `db`, `redis`, `migrate`, `app`, `celery-worker`, and `celery-beat` all
come up together, in the right order: `app`/`celery-worker`/`celery-beat` wait on
Postgres/Redis healthchecks *and* on `migrate` exiting successfully before they start.

## Docker services

| Service | Role |
|---|---|
| `migrate` | One-shot: runs `alembic upgrade head`, then exits. Everything else waits for it. |
| `app` | FastAPI app, `uvicorn --reload`, port `8000` |
| `db` | PostgreSQL 16, named volume `pgdata` |
| `redis` | Celery broker + result backend |
| `celery-worker` | Executes tasks: sending verification emails, token cleanup |
| `celery-beat` | Schedules `cleanup_expired_tokens` hourly |

`migrate` is deliberately a single dedicated service rather than each service
migrating itself on startup вҖ” three containers all running `alembic upgrade head`
concurrently race on creating `alembic_version`, and one loses. Only `migrate` ever
touches the schema; everyone else just waits on `condition: service_completed_successfully`.

View logs for one service: `docker compose logs -f celery-worker`
Rebuild after a dependency change: `docker compose up -d --build`

## Environment variables

| Variable | Purpose | Default |
|---|---|---|
| `ENVIRONMENT` | Free-text environment label | `local` |
| `DB_PASSWORD` | Postgres password (used by the `db` service) | `social` |
| `DATABASE_URL` | Async SQLAlchemy connection string | `postgresql+asyncpg://social:social@db:5432/social` |
| `TEST_DATABASE_URL` | Separate DB used only by the test suite | `postgresql+asyncpg://social:social@db:5432/social_test` |
| `REDIS_URL` | Celery broker/result backend | `redis://redis:6379/0` |
| `SECRET_KEY` | JWT signing secret вҖ” **change in any real deployment** | `change-me-to-a-random-secret` |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | JWT access token lifetime | `30` |

## Database migrations

The `migrate` service runs `alembic upgrade head` automatically on every
`docker compose up` before `app`/`celery-worker`/`celery-beat` start вҖ” a fresh volume
or a full `docker compose down && up` migrates itself, no manual step needed. It's a
no-op if already at head.

```bash
# generate a new migration after changing a model
docker compose exec app alembic revision --autogenerate -m "add comments table"

# review the generated script by hand before committing - autogenerate
# misses some renames/constraint changes

# apply it (or just restart the app container - the entrypoint does this for you)
docker compose exec app alembic upgrade head
```

## Celery

```bash
# confirm the worker is alive and which tasks it registered
docker compose logs celery-worker | grep -A5 "\[tasks\]"

# confirm beat is scheduling the cleanup job
docker compose exec app python -c "from app.tasks.celery_app import celery_app; print(celery_app.conf.beat_schedule)"

# task logs (email sends, cleanup runs)
docker compose logs -f celery-worker
```

In dev, verification emails are printed to the `celery-worker` log instead of sent
over real SMTP вҖ” grep the log for the verification link after registering.

## Running tests

```bash
docker compose exec app pytest -q            # everything
docker compose exec app pytest -q tests/unit          # fast, no DB
docker compose exec app pytest -q tests/integration    # real Postgres test DB
```

Tests run against `TEST_DATABASE_URL` (a separate database on the same Postgres
server), created automatically on first run. Every test gets a fresh schema
(`create_all`/`drop_all` per test) вҖ” no shared state between tests.

## API examples

```bash
# 1. Register
curl -X POST http://localhost:8000/api/v1/auth/register \
  -H "Content-Type: application/json" \
  -d '{"username":"alice","email":"alice@example.com","password":"supersecret"}'

# 2. Verify (grab the link from `docker compose logs celery-worker`)
curl "http://localhost:8000/api/v1/auth/verify-email?token=<token-from-log>"

# 3. Login
curl -X POST http://localhost:8000/api/v1/auth/login \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "username=alice@example.com&password=supersecret"
# -> {"access_token": "...", "token_type": "bearer"}

# 4. Create a post
curl -X POST http://localhost:8000/api/v1/posts \
  -H "Authorization: Bearer <access_token>" \
  -H "Content-Type: application/json" \
  -d '{"content":"hello world"}'

# 5. Feed of people you follow
curl http://localhost:8000/api/v1/feed -H "Authorization: Bearer <access_token>"

# 6. Search & paginate
curl "http://localhost:8000/api/v1/posts?q=hello&page=1&size=20&sort=newest"
```
